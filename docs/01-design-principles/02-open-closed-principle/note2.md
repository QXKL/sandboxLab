## 现代的优质项目都符合SOLID原则吗？
**并非所有现代优质项目都严格符合 SOLID 原则。** 

事实上，随着近年来开发范式、微服务架构以及函数式编程（FP）的兴起，现代软件工程对 SOLID 原则的态度已经从“奉为圭臬”转变为“辩证吸收”。

---

### 1. SOLID 是面向对象（OOP）的产物

SOLID 原则最初由罗伯特·C·马丁（Uncle Bob）在 2000 年代初提出，它是深度绑定面向对象编程（OOP）的。

* **现状：** 现代优质项目大量采用**多范式混合编程**。诸如 Go、Rust、TypeScript 等现代语言更倾向于组合优于继承（Composition over Inheritance）以及函数式编程思想。
* **影响：** 像 **L (里氏替换原则)** 和 **I (接口隔离原则)** 在没有深层继承树或不依赖传统类接口的项目（如纯 Go 或 React 函组件项目）中，其实用性会大大减弱，甚至没有用武之地。

---

### 2. 现代框架和微服务削弱了部分原则

现代项目的架构重心从“单体类的设计”转移到了“服务间的设计”或“组件间的设计”。

| 原则 | 传统理解 | 现代优质项目的权衡（以微服务/前端为例） |
| --- | --- | --- |
| **S** (单一职责) | 一个类应该只有一个发生变化的原因。 | 依然被奉为真理。但在微服务中，它演变成了“微服务内聚”；在 React/Vue 中，演变成了“单一功能组件”。 |
| **O** (开闭原则) | 对扩展开放，对修改关闭（通过继承或接口）。 | 现代项目更崇尚 **KISS 原则（Keep It Simple, Stupid）**。与其为了未来可能不需要的扩展去设计复杂的抽象，不如写出最简单的代码，等到需求变了直接修改（甚至重写），只要有完善的自动化测试保障即可。 |
| **D** (依赖倒置) | 高层模块不应依赖低层模块，二者都应依赖抽象。 | 在后端（如 Spring Boot、NestJS）中通过 **DI（依赖注入）** 依然非常流行。但在serverless、云原生函数或轻量级脚本中，这种抽象往往显得过于笨重。 |

---

### 3. 过度设计的代价：复杂性 vs. 可读性

现代工程越来越意识到：**抽象是有成本的。**
为了完全符合 SOLID 原则（尤其是 O 和 D），开发者往往需要引入大量的接口、抽象类和工厂模式。

> **过度设计的后果：** 代码库变得极度碎片化。追踪一个简单的业务逻辑，需要在 7 个不同的文件、接口和实现类之间反复跳转。

现代优质项目（例如很多知名的开源项目：Go 标准库、Tailwind CSS、乃至高性能的 Linux 内核/Redis 等 C 语言项目）更看重**代码的直观性、可读性以及运行效率**。如果引入 SOLID 会导致系统复杂性暴增，现代架构师会果断地选择放弃。

---

### 4. 现代优质项目更看重什么？

比起 20 年前的 SOLID，现代软件工程沉淀出了更契合当下高并发、快迭代、云原生环境的新原则。

**Cupid 原则：** 近年由 Dan North 提出，用来替代 SOLID。它更强调：
* **C**omposable（易于组合）
* **U**npredictability-friendly（拥抱不确定性/易于修改）
* **P**redictable（行为可预测）
* **I**diomatic（符合语言习惯）
* **D**omain-driven（领域驱动）


* **KISS & DRY：** 保持简单（Keep It Simple, Stupid）和不要重复自己（Don't Repeat Yourself）。
* **可测试性（Testability）：** 代码是否容易写单元测试。只要代码能轻松被测试覆盖，其解耦程度通常就已经足够好，不需要刻意用 SOLID 去硬套。

### 总结

**SOLID 是工具，而不是信仰。** 现代的优质项目不会盲目打勾去满足每一个原则。  
它们通常会在 **S（单一职责）** 上保持严苛，在 **D（依赖倒置）** 上保持理性，而在 **O、L、I** 上根据具体的编程范式和业务场景做出灵活的妥协。

---

## 就比如说，使用抽象接口编程，不一定比直接写两个实现类更好？

这是现代软件工程中一个非常经典的痛点：**面向接口编程的“神话”**。

答案是肯定的：**在很多情况下，直接写两个实现类（甚至直接写一个类，需要时再重构），确实比盲目引入一个抽象接口更好。**

为了让你更直观地看清这两者的权衡，我们可以从以下几个维度来对比：

---

### 1. 接口不是免费的（抽象的成本）

在很多人的潜意识里，接口只是一行 `interface` 代码，成本为零。但实际上，引入接口会带来**认知负载**和**维护成本**：

* **追踪成本增加：** 当你按住 Ctrl 点击一个方法时，IDE 会跳转到接口定义，而不是具体的实现。你必须再多点一次，从列表中选择进入某一个具体的实现类。如果套了多层接口，代码的“可读路径”就会被严重割裂。
* **修改成本翻倍：** 如果你发现某个方法需要增加一个参数，你不仅要改实现类，还必须去修改接口定义。如果这个接口被多个地方引用，改动范围会迅速扩大。

---

### 2. 接口的真正价值是什么？

现代开发中，我们只有在明确需要接口带来的**核心价值**时，才会去写接口。这个价值通常只有两个：

1. **多态切换（运行时动态扩展）：** 比如你的系统需要同时支持 `MySQL` 和 `PostgreSQL`，且在运行时根据配置切换。
2. **解耦与边界（尤其是跨团队/跨模块）：** 比如基础架构团队写了一个公共组件，他们提供接口给业务团队，业务团队不需要关心底层怎么实现。

**反观你的例子：** “直接写两个实现类”。
如果这两个实现类在业务中**从来不会被同时、动态地替换**（例如：一个是 `ProdUserService`，另一个是 `MockUserService` 仅用于单元测试），那么在现代开发中，我们有更好的替代方案：

* **对于测试：** 现代测试框架（如 Mockito、GoMock、Jest 等）已经足够强大，它们可以直接 Mock 具体的类、函数甚至私有方法，**根本不需要为了写单元测试而刻意抽出一层接口**。

---

### 3. 场景对比：什么时候该用哪个？

我们可以用一个简单的场景来看看两种做法的优劣：

#### ❌ 盲目遵守 SOLID 的做法（过度设计）

你定义了一个 `OrderProcessor`（订单处理）接口，然后写了 `DomesticOrderProcessor`（国内订单）和 `InternationalOrderProcessor`（国际订单）。

```
                  ┌──────────────────┐
                  │ <interface>      │
                  │  OrderProcessor  │
                  └────────┬─────────┘
                           │
          ┌────────────────┴────────────────┐
          ▼                                 ▼
┌──────────────────────────┐    ┌─────────────────────────────┐
│ DomesticOrderProcessor   │    │ InternationalOrderProcessor │
└──────────────────────────┘    └─────────────────────────────┘

```

* **代价：** 国内订单和国际订单的业务流程、所需参数可能完全不同。强行让他们实现同一个接口，会导致接口中出现大量 `UnsupportedOperationException`，或者接口参数变成一个巨大的、什么都装的 `Map`。这就违反了 **I（接口隔离原则）**。

#### 现代务实的做法（YAGNI 原则）

> **YAGNI:** You Ain't Gonna Need It（你不需要它——除非当前真正需要，否则不要预测未来）。

直接写两个独立的类：`DomesticOrderService` 和 `InternationalOrderService`。它们之间没有任何继承或接口关系。

* **优势：** 两个类的代码高度内聚，国内业务怎么改都不会影响国际业务。如果未来某天，系统真的出现了“需要统一批量处理各类订单”的需求，此时再通过 IDE 的 `Refactor -> Extract Interface` 功能，**一键自动生成接口**。

---

### 总结

现代优质项目更倾向于 **“演进式架构”**。

在刚开始写代码时，**直接写实现类是“显式”的、直观的**。只有当具体的实现类超过两个，且它们之间确实存在强烈的共性、需要被动态组合时，才去重构出接口。

正如极限编程（XP）推崇的：**先让它工作，再让它优雅，最后让它高效（Make it work, make it right, make it fast）。** 别在一开始，就被接口绑架了。

## 问一下模板方法模式还在使用吗？

依然在大规模使用。
只要是做标准工作流、流程引擎、或者生命周期管理的，基本全逃不出模板方法模式。比如：
- Spring 框架里的 JdbcTemplate / RestTemplate： 帮你把“打开连接、处理异常、关闭连接”的骨架写好了，你只需要传进去一个具体的 SQL 或 RowMapper（细节）。
- 前端微应用/生命周期钩子： 框架规定好了组件的加载流程（Init -> Render -> Destroy），你在子组件里重写的 ngOnInit 或 mounted()，本质上就是被模板方法调用的细节。

### 务实避坑提示
模板方法模式有一个现代开发者很讨厌的副作用：它依赖继承（Inheritance）。
如果流程非常复杂，会导致子类继承了臃肿的父类，违反了“组合优于继承”的现代原则。

所以，现代优质项目在实现这个模式时，往往会把它和策略模式结合。父类不再是 abstract 的，而是把那些变化的步骤（比如 calculateFee）做成接口（策略），通过组合的方式传进来。这也就是著名的 “好莱坞原则”：Don't call us, we'll call you（别调用我，我会调用你）。
