# 微服务架构 - 测试题

## 一、选择题

### 1. 微服务架构的核心特征不包括？
A. 按业务能力组织  
B. 独立部署  
C. 共享数据库  
D. 去中心化数据管理

**答案**：C

**解析**：微服务架构强调去中心化数据管理，每个服务有自己的数据库，而不是共享数据库。

---

### 2. 微服务间通信的主要方式是？
A. 直接访问数据库  
B. REST API或消息队列  
C. 共享内存  
D. 文件系统

**答案**：B

**解析**：微服务间通过轻量级机制通信，通常是REST API（同步）或消息队列（异步）。

---

### 3. 微服务架构解决分布式事务的常用方案是？
A. 两阶段提交  
B. Saga模式或最终一致性  
C. 全局锁  
D. 单体事务

**答案**：B

**解析**：微服务中常用Saga模式（补偿事务）或基于事件的最终一致性来处理分布式事务。

---

### 4. 以下哪个不是微服务的优点？
A. 独立部署  
B. 故障隔离  
C. 降低系统复杂度  
D. 技术栈灵活

**答案**：C

**解析**：微服务会增加系统复杂度（分布式系统复杂性），而不是降低复杂度。

---

### 5. 微服务拆分应该避免？
A. 按业务能力拆分  
B. 过度拆分  
C. 高内聚低耦合  
D. 单一职责

**答案**：B

**解析**：应避免过度拆分，太细粒度会增加复杂度和维护成本。

---

## 三、判断题

### 1. 微服务架构适合所有类型的应用。（ ）

**答案**：✗

**解析**：微服务不适合小型简单应用，会带来不必要的复杂度。

---

### 2. 微服务架构中，所有服务应该使用相同的技术栈。（ ）

**答案**：✗

**解析**：微服务的优点之一是技术栈多样性，可以根据需求选择合适的技术。

---

### 3. 微服务架构可以实现故障隔离。（ ）

**答案**：✓

**解析**：某个服务故障不会影响其他服务，这是微服务的重要优点。

---

### 4. 微服务架构中，服务间可以直接访问彼此的数据库。（ ）

**答案**：✗

**解析**：服务间应该通过API通信，不应该直接访问其他服务的数据库。

---

### 5. 微服务架构比单体应用更容易测试。（ ）

**答案**：✗

**解析**：微服务的集成测试和端到端测试更复杂，需要启动多个服务。

## 四、简答题

### 1. 比较单体应用和微服务架构的优缺点。

**答案**：

**单体应用**：

**优点**：
- 开发简单：所有代码在一起
- 部署简单：一个应用包
- 测试简单：集成测试容易
- 事务管理简单：本地事务
- 性能好：进程内调用

**缺点**：
- 扩展困难：只能整体扩展
- 部署风险高：任何改动都要重新部署
- 技术栈固定：难以引入新技术
- 故障影响大：一个模块问题影响全局
- 代码耦合：模块间耦合严重

---

**微服务架构**：

**优点**：
- 独立部署：服务独立发布
- 故障隔离：服务故障不影响其他服务
- 技术栈灵活：可以使用不同技术
- 团队独立：团队可以独立工作
- 易于扩展：按需扩展特定服务

**缺点**：
- 复杂度高：分布式系统复杂
- 运维困难：需要管理多个服务
- 分布式事务：数据一致性难保证
- 服务间通信：网络延迟和故障
- 测试复杂：集成测试困难

---

### 2. 说明微服务架构中如何处理分布式事务。

**答案**：

**方案1：Saga模式（补偿事务）**

原理：将分布式事务拆分为多个本地事务，失败时执行补偿操作。

```java
public void createOrder(Order order) {
    try {
        // 1. 创建订单
        orderService.create(order);
        
        // 2. 扣减库存
        inventoryService.decrease(order.getItems());
        
        // 3. 扣减余额
        paymentService.deduct(order.getAmount());
        
    } catch (Exception e) {
        // 补偿事务（回滚）
        inventoryService.increase(order.getItems());
        orderService.cancel(order.getId());
        throw e;
    }
}
```

---

**方案2：最终一致性（事件驱动）**

原理：通过异步事件保证最终一致性。

```java
// 1. 订单服务：创建订单
public void createOrder(Order order) {
    orderRepository.save(order);
    eventPublisher.publish(new OrderCreatedEvent(order));
}

// 2. 库存服务：监听事件
@EventListener
public void onOrderCreated(OrderCreatedEvent event) {
    inventoryService.decrease(event.getItems());
}

// 3. 支付服务：监听事件
@EventListener
public void onOrderCreated(OrderCreatedEvent event) {
    paymentService.deduct(event.getAmount());
}
```

---

**方案3：TCC（Try-Confirm-Cancel）**

原理：两阶段提交的变种。

---

**选择建议**：
- 强一致性要求低 → 最终一致性
- 需要补偿 → Saga模式
- 强一致性要求高 → 避免分布式事务（重新设计）

---

### 3. 微服务拆分的原则是什么？请举例说明。

**答案**：

**原则1：单一职责原则**

每个服务只负责一个业务能力。

```
✅ 好的拆分：
- 用户服务：注册、登录、个人信息
- 订单服务：下单、查询订单、取消订单

❌ 不好的拆分：
- 用户订单服务：用户 + 订单混在一起
```

---

**原则2：高内聚低耦合**

服务内部功能紧密相关，服务间松耦合。

```
✅ 高内聚：
订单服务内部功能都与订单相关：
- 创建订单
- 查询订单
- 取消订单
- 订单状态变更

✅ 低耦合：
订单服务通过API调用用户服务，不直接访问用户数据库
```

---

**原则3：避免过度拆分**

不要拆分得太细。

```
❌ 过度拆分：
- 用户注册服务
- 用户登录服务
- 用户信息服务
- 用户密码服务

✅ 合理拆分：
- 用户服务（包含注册、登录、信息、密码）
```

---

**原则4：按业务能力拆分**

按业务领域拆分，而不是按技术层次。

```
✅ 按业务拆分：
- 用户服务
- 商品服务
- 订单服务
- 支付服务

❌ 按技术拆分：
- 数据库服务
- 缓存服务
- 消息服务
```

---

**示例：电商系统拆分**

```
核心服务：
- 用户服务：用户注册、登录、个人信息
- 商品服务：商品目录、搜索、详情
- 订单服务：下单、订单查询、订单管理
- 支付服务：支付、退款
- 库存服务：库存管理、库存扣减
- 物流服务：发货、物流追踪

支撑服务：
- 通知服务：短信、邮件、推送
- 营销服务：优惠券、活动
- 搜索服务：商品搜索、推荐
```

---

💡 **提示**：微服务架构适合大型复杂系统，小型应用使用单体架构即可！
